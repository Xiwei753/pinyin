# Android 九键输入法性能重构技术路线

这份文档只回答一件事：为什么现在会卡，以及下一步应该怎么重构。结论先说：当前问题不是小 bug，也不是再调几条 SQL 或多加几层缓存就能解决的问题，而是输入事件热路径设计错了。

## 1. 当前结论

- 当前 Android 九键输入法的性能问题已经是架构问题，不是单点实现问题。
- 输入热路径不允许阻塞，按键、震动、preedit 更新必须立刻完成。
- 候选生成必须从按键事件链路中拆出去，变成后台异步路径。
- 继续在现有同步链路里堆 cache / if / score，只会让代码更复杂，不会从根上解决卡顿和丢事件。

## 2. 当前错误架构

当前链路的核心问题是：按一次键就同步做一整套重计算。

- `T9Engine.getCandidates()` 在 buffer 变化时同步调用 `generateCandidates()`。
- `generateCandidates()` 内部会做 composition 遍历、词库查询、DP 动态拼接、排序。
- `getSentenceCandidates()` 仍然会构造大量 `partStr` 并做动态组合。
- 候选日志输出和 UI 刷新也会叠加耗时。
- 这些操作都不应该发生在按键热路径里。

直白地说：现在每次按键都像“先跑一段小型离线计算，再决定要不要回显按键”。这会直接导致主线程被占住。

## 3. 为什么会卡、为什么会丢拼音/丢震动、为什么不是 UI 小问题

- **为什么会卡**：按键后同步执行候选重计算，CPU 时间被候选逻辑吃掉，输入事件处理和 UI 帧更新排队。
- **为什么会丢拼音**：新按键到来时，前一个按键触发的重计算还没结束，buffer/preedit 的更新节奏被拖慢或被后续状态覆盖，用户看到的就是“少字母/少音节”。
- **为什么会丢震动**：haptic 如果放在候选计算后，或和重计算共享阻塞链路，快速输入时就会出现触发延迟甚至漏触发。
- **为什么不是 UI 小问题**：UI 渲染只是结果展示层，真正阻塞发生在输入到候选生成的热路径。只改候选栏样式、动画或 View 结构不能解决根因。

## 4. 正确目标架构

社区成熟输入法的共识是：前端热路径极轻 + 后台候选引擎 + 预编译词库索引。不是每次按键同步跑 SQLite/DP。

### Key Input Hot Path

- 接收按键。
- 立即更新 buffer。
- 立即触发 haptic。
- 立即更新 preedit。
- 不等待候选生成。
- 不同步查 SQLite。
- 不跑 DP。

### Candidate Worker Path

- 后台线程/单线程 worker 处理候选。
- 每次输入生成 `requestId`。
- 老 request 自动丢弃。
- 只把最新 request 的候选回传 UI。
- UI 主线程只做轻量渲染。

### Dictionary Index Path

- 构建期生成预编译索引。
- 运行时优先查内存索引或 mmap 索引。
- SQLite 只能做冷启动、调试或兜底，不应成为每键候选主路径。

### UI Path

- 候选栏刷新不能影响按键。
- haptic 必须在候选计算前触发。
- preedit 必须在候选返回前就更新。

## 5. 词库索引路线

### 方案一：继续 SQLite

- 优点：实现简单。
- 缺点：每键查询和多组合查询容易卡，尤其在 IME 热路径里不合适。
- 结论：不作为主路线。

### 方案二：Kotlin 内存索引

- 优点：改动较小，容易验证。
- 缺点：内存压力可能变大。
- 结论：可做短期过渡。

### 方案三：预编译 native/mmap 索引

- 优点：启动后查询快、适合大词库、接近成熟输入法路线。
- 缺点：实现工作量更大。
- 结论：长期主线。

建议路线：先做“轻量 Kotlin 内存索引 V1”，跑通异步候选链路并验证实机手感，再升级到“mmap/native index V2”。

## 6. 候选分层路线

必须坚持以下规则，不允许回退：

- 九键数字 -> 拼音音节解码 -> 拼音 preedit -> 拼音查字词。
- 不能回退到数字 prefix 直接查中文词库。
- 完整拼音整词命中优先。
- 动态拼接只能兜底。
- `base.dict.yaml` 做常用词语词库。
- `8105.dict.yaml` 只做单字候选表。
- 不能把 `8105.dict.yaml` 当主词库刷冷僻字。
- 候选栏宁可少，不要乱喷垃圾候选。

## 7. 分阶段实施计划

### 阶段 A：止血

- 按键热路径不再等待候选生成。
- haptic 在按键处理最前面触发。
- preedit 立即更新。
- 候选异步返回。
- 老候选可以短暂显示，但不能阻塞输入。
- 加性能指标：每次按键主线程耗时必须低于 5ms，候选返回目标低于 30ms。

### 阶段 B：候选 worker

- 新建 `CandidateRequest` / `CandidateResult`。
- 每次输入递增 `requestId`。
- 后台单线程 worker 计算候选。
- 新输入到来取消或丢弃旧 request。
- UI 只接受最新 request。
- 测试快速输入不会丢 buffer、不会丢 haptic。

### 阶段 C：词库索引 V1

- 启动后从 SQLite 或 asset 加载常用索引到内存。
- 建立 `pinyinExactIndex`、`singleSyllableIndex`、`phraseIndex`。
- 查询时不再频繁 SQLite。
- 保留 SQLite 作为构建产物或兜底，不作为热路径主查询。

### 阶段 D：词库索引 V2

- 设计构建期二进制索引。
- 支持 mmap 或 native 查询。
- Kotlin/Android 只拿结果，不负责大规模遍历。
- 为未来 Linux 端共享 engine 做准备。

### 阶段 E：清理旧补丁

- 清理 `T9Engine` 里大量 magic score。
- 清理过度 DP 拼接。
- 清理同步查询。
- 清理只为过测试加的 workaround。
- 建立性能基准测试。

## 8. 验收标准

- 快速输入 `288249464` 不丢数字、不丢 preedit、不丢震动。
- 快速输入 `546842692674264` 不丢 preedit。
- 按键后 preedit 立即变化，不等待候选。
- 候选可以晚一点出现，但不能阻塞按键。
- 每次按键主线程耗时目标 < 5ms。
- 候选生成目标 < 30ms，慢候选不能阻塞下一次输入。
- haptic 触发必须发生在候选计算之前。
- 候选点击必须提交当前 UI 缓存候选，不重新全量计算。

## 9. Golden Cases（必须保留）

- `96` -> preedit `wo`，候选包含“我”
- `64` -> preedit `ni`，候选包含“你”
- `82` -> preedit `ta`，候选包含“他/她”
- `33` -> preedit `de`，候选包含“的/得/地”
- `744` -> preedit `shi`，候选包含“是”
- `28` -> preedit `bu`，候选包含“不”
- `28824` -> preedit `bu tai`，候选包含“不太”
- `288249464` -> preedit `bu tai xing`，第一候选必须是“不太行”
- `288249464` 不应该出现“不太新股”“不太英语”
- `546842692674264` -> preedit `jin tian wan shang`，候选包含“今天晚上”

## 10. 明确不要做

- 不要写实现代码。
- 不要继续给 `T9Engine` 加缓存补丁。
- 不要继续调 magic score。
- 不要继续加 SQLite 查询优化补丁。
- 不要改 UI。
- 不要加同步。
- 不要加 AI。
- 不要加云词库。
- 不要做 x86 Android 特殊适配。
- 不要回退到数字直接查词。
- 不要只写空泛文档，必须能指导下一步重构。
